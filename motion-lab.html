SCROLL STAGGER
Internal · not shipped · 2026

Stagger that waits for the scroll.

Right now below-fold elements fire as soon as they peek past the bottom edge, then run a 1.56s settle. So they start, and often finish, while still parked at the foot of the screen, before you've scrolled them into view. The motion outruns the reader.

Seven techniques below, all on the existing tokens. The same Title / Description / Buttons block in each, so only the motion differs. Scroll each one: the specimen sits well below the fold so you feel the trigger point, not just see it. Every block names its difference up top.

  1. A Scroll-driven, per item. Each element on its own scroll timeline.
  2. B Observer, deeper trigger. The safe fix to what ships, works everywhere.
  3. C Index entrance on load. For above-fold heroes, no scroll to wait on.
  4. D Scroll-driven, group sweep. One shared timeline, the whole block as a unit.
  5. E Clip-path wipe. Text wipes up behind a moving edge.
  6. F Word by word, in four speeds (F1–F4), slow to snap. The deck resolves a word at a time.
  7. G Focus-pull. Blur to sharp, nothing moves.
Recommendation: A or D for the homepage (D if you want the block to land as one move, A if each row should earn its own arrival), B everywhere scroll-driven isn't supported yet. A, B and D fix the real bug, motion that outruns the reader. C handles above-fold heroes. E, F and G are the same fix dressed in different materials: pick one as the house feel. All replace the .d1–.d4 ladder with a single --stagger index.
Now · current behaviour

Fires at the edge.

Roughly what ships: triggers near the bottom edge (rootMargin −8%) on the long 1.56s settle. Watch it animate before it's really in view.

The bug Fires too early, lasts too long. Everything below is a way out of this.

A · recommended

Scroll-driven, view-timeline.

Progress is bound to the element's travel through the viewport via animation-timeline: view(). It resolves in the reading band, and scrolling back up plays it in reverse. No JS. --stagger offsets each item's range for the cascade.

vs D Each element gets its own scroll timeline, so in a tall block they can arrive far apart. Incremental, not choreographed.

B · safe everywhere

Observer, deeper trigger.

Same IntersectionObserver we ship, fired deep into the viewport (rootMargin bottom −28%) on a snappier 0.62s settle, with an index --stagger delay. It lands in reading position.

vs A Same look, but JS-driven and one-shot (no reverse on scroll up). The choice when scroll-driven CSS isn't supported.

C · above-fold heroes

Index entrance, on load.

The brief's pattern: no scroll to wait for, so bind the cascade to load and drive the delay off the --stagger index. Named to match the house .reveal.

D · recommended (group)

Scroll-driven, group sweep.

One view-timeline named on the container, sliced per --stagger index. The whole block reveals as a single coordinated move the moment it enters, instead of each item firing on its own. No JS.

vs A Shared timeline, not one per item. The block lands as a unit and stays tight however tall it is. The cinematic option.

E · material

Clip-path wipe.

Each line is revealed behind a moving edge with clip-path, so the text wipes up into place rather than sliding in. Editorial, typographic. Observer-triggered deep, index --stagger.

vs B Same trigger, different material: a hard reveal edge instead of a translate. The text uncovers rather than moves.

G · restraint

Focus pull.

No translate, no wipe. Each item resolves from soft to sharp in place, like a lens finding focus. Calm where a rise can read busy. Observer-triggered deep, index --stagger.

vs all others Nothing moves. Blur to sharp only. The quietest reveal here, and the hardest to overdo.

Porting to the real motion system. A, B and D all retire the .d1–.d4 classes for one --stagger index set per child, which scales past 4 items for free. A and D are @supports (animation-timeline: view()) blocks in motion.css with nothing in motion.js; D adds view-timeline-name on the container. B is a three-line change to the observer in motion.js (rootMargin bottom to −28%, threshold to 0, the per-item duration token) plus an index delay. E, F and G are pure material swaps: the same trigger as B with a different keyframe, so they layer on once the trigger is fixed. New timing values (--t-stagger-item, the range percentages) go in tokens.css first, per the design-system rule.
